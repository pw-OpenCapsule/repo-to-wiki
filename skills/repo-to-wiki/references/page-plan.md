# 多子页结构规划

产出一棵「索引页 + 若干子页」的树，挂在飞书 wiki「各项目知识库 / 对应项目组」下。**业务理解放最前**（新人先看业务再看源码）。

## 固定骨架

```
📁 <项目名> 源码理解        ← 索引页：一句话定位 + 关键数字表 + 子页目录 + hero 图
├─ 🏁 业务理解              ← 置顶推荐：替谁解决什么生意问题、业务对象、业务域、业务场景、端到端业务流(画板)
├─ 架构总览                 ← 分层（来自 /understand 的 layers）+ 请求/调用生命周期(画板) + 技术栈
├─ 核心流程                 ← 最该看懂的主链路：入口 → ... → 出口(画板)
└─ <按项目补>              ← 子系统各一页（如模板系统 / 算法 / 存储部署 / 接口层），有就拆，没有就并
```

子页数量看项目大小：小项目 索引+业务+架构+流程 4 页够；大项目按子系统拆到 6–8 页。

## 每页的最小要求

- 顶部 1 张插图（hero / 概念图，固定烂图风、内容正确）。
- 关键流程用 1 张 `<whiteboard type="mermaid">` 原生画板（不是代码块）。
- 正文人话优先：表格 > 嵌套 bullet；标注真实「现状坑」（空实现、被注释掉的、设计文档里只是「建议」未落地的）。
- 索引页：关键数字表（文件数 / 分层数 / 等）+ 子页目录链接 + 业务理解用高亮框置顶。

## 内容来源映射

| 页面 | 主要数据来源 |
|---|---|
| 索引 | /understand 的 project + layers 概览 |
| 业务理解 | domain-analyzer 的 domain-graph.json（业务域/业务流/步骤） |
| 架构总览 | /understand 的 layers + 入口文件 |
| 核心流程 | codegraph trace 主链路 + 直接读核心文件 |
| 上下游（可选） | code-search 的跨仓调用关系 |
| 子系统页 | knowledge-graph.json 对应模块 + 直接读 |

## 语气纪律（避免 AI 味）

- 个人化 hedging（「我自己的习惯」「navify 这次就是…」），别派一堆结论。
- 砍 emoji + callout 堆砌；自然 h2 标题，别 Part 1/2/3。
- 没出处的数字别编（不写「省 30%」这种）。
- 真实例子直接放正文，不用 callout 包。
